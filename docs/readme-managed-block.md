# README managed block — a devkit-owned region inside a user-editable README

**Status:** backlog / design-only (task #9). Nothing built. Makes the generated brain's
`README.md` a **hybrid**: a devkit-owned region the upgrader keeps current, wrapped by
user-owned space the upgrader never touches.

---

## 0. The problem (observed 2026-07-07)

The generated brain's `README.md` is **devkit-owned**: `update_brain.py` re-emits it from
`template/` **wholesale** (it is in the re-emit list, *not* the `PRESERVE` list of
`vault/ data/ config/ CLAUDE.md GEMINI.md`). So any edit a user makes to the brain README
is **silently clobbered** on the next `update_brain --apply`.

This bit us for real: a `~/second-brain` README had a hand-added "planned retrieval-quality
summary" — which was both (a) misplaced devkit-roadmap content, and (b) fated to be
overwritten anyway. The lesson isn't only "that content was misplaced"; it's that the brain
README has **no place for the user at all**, which is surprising for the front-page doc of
a repo they own.

## 1. The design — a marked managed block

Wrap the devkit's README content in HTML-comment markers; let the user own everything
outside them:

```markdown
<!-- Personal notes, links, TODOs — the user owns this; the devkit never touches it. -->

<!-- BEGIN generated by second-brain-devkit — do not edit inside; regenerated on update -->
… the devkit's operational README (Setup → Everyday use → Layout → …) …
<!-- END generated by second-brain-devkit -->

<!-- More user space below, equally untouched. -->
```

- **Devkit content lives *between* the markers; the user writes *above or below*.**
- **Create and update unify.** The `template/` README already ships **with** the markers
  around the devkit body. Initial create copies the file whole (user regions empty). A
  later `update_brain` **splices** the fresh devkit body between the *existing* file's
  markers, preserving the user's preamble/appendix byte-for-byte.
- **Opt-out by deleting the start marker.** If `BEGIN` is absent (the user removed it to
  take full ownership), the upgrader implants **nothing** and leaves the README alone —
  the same "detect + instruct, never force" stance as `install_skill.py` / `doctor.py`.
- **Markers are inert.** HTML comments don't render in Markdown / Obsidian, so there's no
  visible clutter.

This is the devkit's established **managed-block pattern**, generalized to the brain
README — the same shape as `install_skill.py --nudge`'s marked block in `~/.claude/
CLAUDE.md`, the `ai-project-status` block in the devkit's own `CLAUDE.md`, and the
`related_auto:` block in the [auto-linking design](auto-linking.md) (task #8).

## 2. Contract

- Editing **inside** the managed region is **not** preserved — that region is the devkit's,
  and `update_brain` overwrites it. The `BEGIN` comment says so inline. User content goes
  **outside** the markers.
- `update_brain` still reports the README under NEW / CHANGED / preserved so a dry-run
  shows exactly what the splice would change.

## 3. Implementation sketch

- **Golden first** (per the build loop): add the markers to the golden `README.md`, then
  `vendor_golden.py` → rebuild `template/`. The markers are just comments, so the
  structural-diff oracle and the brain's `self_test` are unaffected (still byte-exact).
- **`update_brain.py`**: replace the wholesale README copy with a splice —
  `existing.preamble + START + template.body + END + existing.appendix`. On a target whose
  README lacks `BEGIN`/`END`, skip it (leave user-owned) and say so in the report.
- **Guard the malformed case**: `BEGIN` without `END` (or vice-versa) → skip + warn rather
  than guess a boundary.
- **`create_second_brain.py`** is unchanged — it copies the template README verbatim, which
  already contains the markers with empty user regions.

## 4. Edge cases

- **No markers** (user deleted them) → devkit hands off entirely.
- **Only one marker present** → skip + warn (don't guess).
- **User edited inside the region** → overwritten by design; the inline notice + dry-run
  preview are the mitigation.
- **Multiple BEGIN/END pairs** → v1 supports exactly one managed region; reject extras.
